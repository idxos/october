Terminology

Within the API documentation we use the following terminology:

API
Application programming interface - the mechanism by which dealers can interact with our system. The architecture for sharing content and data between web applications.
REST
Representational state transfer - the API HTTP methods standards by which we follow.
JSON
JavaScript Object Notation - the language independent data format that we primarily use to transmit data between servers and web applications.
Content Negotiation
Content negotiation is the client telling the server what it is sending and what it wants in return, and the server determining if it can do what the client requests.
Collection
A addressable set of entities. Typically, all entities contained in the collection are of the same type, and share the same base URI as the collection. Collections only allow GET (fetch a list) operations. Because our collections support pagination, the API will limit to a set number of items per "page" of results. For example, in the URL: api.wpswebservices.com/vendor, the vendor is the collection or sometimes referred to as the "service".
Entity
An addressable item being returned. Entities are distinguished by a unique identifier present in the URI. For example, in the URL: api.wpswebservices.com/item/28-1111, the 28-1111 part is an item identifier which we consider the "entity". Entities allow GET (fetch an entity) operations.
Relational Links
A URI to a resource that has the described relation. Relational links allow you to describe relations between different entities and collections, as well as directly link to them so that the web service client can perform operations on those relations. These are also sometimes called hypermedia links.

REST services return entities and collections, and provide hypermedia links between related entities and collections. Resource objects coordinate operations, and return entities and collections.
